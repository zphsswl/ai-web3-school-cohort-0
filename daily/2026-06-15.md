### Handbook Key Concepts

- **Agent Workflow 第一性原理**: 高风险 Agent 不能只有"下一步推理"，必须有状态、边界和停止条件
- **Task Graph**: 把 "帮我做一次低风险 swap" 拆成 9 步可验证节点，而不是一句话让 Agent 自由发挥
- **State Machine**: Agent 状态不只有聊天历史 → draft → plan_ready → simulation_failed → waiting_user_confirmation → submitted → confirmed
- **Human-in-the-loop 分层**: 只读自动执行 / 小额白名单 session key / 高风险必须确认 / 超出 policy 直接拒绝
- **Web3 Tool Use 读写分离**: 不要给 Agent "万能 RPC" 工具 — 读和写必须是不同工具、不同权限、不同审计日志

## 🔍 Questions / Blockers

- [ ] Intent Parser 在架构中的位置：应该作为 Task Graph 的第一个节点（Agent 内部），还是作为独立的前置服务？
- [ ] 链上上下文依赖：Parser 解析「帮我 swap」时需要知道用户余额吗？如果需要，这是 read-only 依赖，应该在 Parser 层还是 Policy Check 层做？
- [ ] LLM 幻觉风险：用户说「帮我把 PEPE swap 成 ETH」— LLM 可能拼出正确合约地址，但也可能幻觉。合约地址验证应该在 Parser 层还是 Policy Check 层？
- [ ] 中英混合输入的 schema 设计：`TransactionIntent` 的字段名是英文，但用户输入是中英混合 — 是否需要双语 field mapping？
