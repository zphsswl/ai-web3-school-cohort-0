### 🟢 Minimum Path (30 min)

- [ ] **速读 [Agent Identity](https://aiweb3.school/zh/handbook/bridge/agent-identity/) — Agent Profile + Service Endpoint 节点** (10 min)
  - Agent Profile = 公开说明文件：名称、描述、服务范围、能力列表、owner、版本
  - 核心原则：Profile 要同时给人和机器看 — 人类理解 Agent 做什么，机器解析 endpoint/capabilities/schemas
  - Service Endpoint 是外部调用入口：HTTPS API / A2A / MCP server / Webhook
- [ ] **写最小 README — 4 个必备段落** (15 min)
  - **What**: AI-native Wallet v0.1 — 用户说自然语言，系统生成结构化交易意图，经过 Policy → Budget → Session Key 安全校验后输出 UserOperation
  - **Why**: AI Agent 管理资金时，不能让 LLM 直接调用转账函数。需要在 Agent 和链之间插一层确定性安全校验
  - **How**: CLI 输入自然语言 → Intent Parser → Permission Policy → Budget Controller → Session Key → Smart Account → UserOperation
  - **Run**: `pip install -r requirements.txt && python cli.py "swap 50 USDC to ETH"`
- [ ] **快速架构草图** (5 min)
  - 用 ASCII 或 Mermaid 画出 6 个组件的流向：User Input → Intent Parser → Policy → Budget → Session Key → UserOp
  - 标注每个组件的安全职责：“Parser: 翻译不执行 / Policy: 规则拦截 / Budget: 额度限制 / SessionKey: 最小权限”

### 🟡 Recommended Path (1 hour)

- [ ] **精读 [Agent Identity](https://aiweb3.school/zh/handbook/bridge/agent-identity/) — 完整 6 个知识节点** (20 min)
  - **Agent Profile**: 同时给人和机器看。更新历史本身是信任信号 — Agent 换了模型/服务端/收款地址都不应该静默发生
  - **Capability**: 不写"我什么都能做"。每个 capability 绑定 schema、价格、限制、风险等级（只读分析=低风险 / 生成交易草稿=中风险 / 自动执行=高风险）
  - **Service Endpoint**: CLI 是当前入口，未来扩展到 API endpoint。Endpoint 更新需要 owner 签名
  - **Registry**: 链上 registry 提供公开可查的身份锚点 — agentId / owner / profile URI / endpoint
  - **Ownership**: 高价值 Agent 不应由单个热钱包控制。operator（运行服务）和 owner（控制身份更新）可分离
- [ ] **参考 [Verifiable AI](https://aiweb3.school/zh/handbook/bridge/verifiable-ai/) — Audit Trail 节点** (8 min)
  - Audit Trail 是最基础、最实用的可验证层：记录输入、输出、模型版本、工具调用、时间、用户确认、交易哈希、错误
  - 核心认知：即使没有 TEE 或 ZK，完整审计轨迹也能支持复盘、争议和改进
  - 应用于本项目：CLI 的终端输出天然就是 trace；REPL 的 `history` 命令就是 audit trail 的交互入口
  - 按风险分层：低风险场景留日志，高风险场景需要 human check

### Handbook Key Concepts

- **Agent Profile 双读者设计**: 人类读描述、价格、owner；机器解析 endpoint、capabilities、schemas、auth。本项目 README 就是这个思路 — 人类读 Overview/Security Model，机器/评审看架构图和代码入口
- **Capability ≠ "我什么都能做"**: 越具体越有用 — swap 能力：「输入 USDC amount + target token + max slippage → 先通过 Policy/Budget 校验 → 用 Session Key 签名 → 返回 UserOp 或拒绝理由」
- **Audit Trail 是最现实的起点**: Verifiable AI 不一定要 TEE/ZK。CLI 的终端输出天然就是 trace — 记录每一步决策：输入了什么、Parser 解析出什么、Policy 是否通过、Budget 剩余多少、Session Key 是否有效、最终是否执行
- **Permission Isolation 已落地**: Parser（只读翻译）≠ Policy（规则判断）≠ Budget（额度控制）≠ Session Key（签名执行）。每个组件只能做一件事，越靠近资产接口越窄
- **Human-in-the-loop 分层**: 对应 Handbook Agent Workflow 的分层确认模型 — 只读自动 / 小额白名单 / 中额确认 / 大额警告 / 越界拒绝。CLI 的确认机制直接实现
