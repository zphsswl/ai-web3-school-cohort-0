**Handbook**: https://aiweb3.school/zh/handbook/tracks/ai-security/

**核心问题**: 当 AI Agent 开始读取文档、调用工具、连接钱包、发起交易时，安全问题就不再只是「模型会不会答错」。真正的问题是：它能看到什么、能调用什么、出错以后谁能发现。

**第一性原理**: 安全的 Agent 不是「更听话的模型」，而是被放进了更小、更清楚、更可审计的行动空间。系统应该通过权限、隔离、校验、模拟、日志和人工确认来限制损失，而不是指望模型永远分得清资料、命令、建议和攻击。

**7 个知识节点**:

| # | 节点 | 难度 | 一句话 |
|---|------|------|--------|
| 1 | **Agent Threat Model** | 🟡 中级 | 画清系统里哪些路径真的危险 — Agent 读什么、调什么、连接了什么账号/资金、哪步执行、哪步确认、出错能否追踪 |
| 2 | **Prompt Injection Defense** | 🟡 中级 | 结构隔离而非 prompt 里的「不要听恶意指令」— 外部资料只能作为 data、检索结果带来源和可信等级、工具调用前检查权限 |
| 3 | **Tool Permission Isolation** | 🟡 中级 | 五层工具权限：只读→用户只读→草稿→写入→资产。越靠后越需要白名单、额度、模拟、人工确认、多签或延迟执行 |
| 4 | **Malicious Document Demo** | 🟢 初级 | 准备一份看起来正常的文档，藏一段越权指令 — 观察系统是否标记外部资料、拒绝越权、不泄露 secret、记录拒绝原因 |
| 5 | **Key / Secret Isolation** | 🔴 高级 | Agent 请求受控动作而非拿秘密本身 — 请求「查询余额」不拿 RPC key、请求「生成交易草稿」不拿私钥、签名由钱包/Smart Account 完成 |
| 6 | **Privacy-preserving Workflow** | 🟡 中级 | 只读必要数据、对外部模型脱敏/摘要、默认隐藏不必要地址和金额、敏感任务优先本地模型/TEE |
| 7 | **AI Behavior Audit** | 🟡 中级 | 审计日志记录原始目标→检索资料→工具调用→输入输出→人工确认→最终结果/失败/拒绝原因 — 事故后知道问题出在哪一层 |

>
> Today explored how AI Agent security goes beyond "make the model more obedient" — it's about constraining the action space through permissions, isolation, simulation, logging, and human confirmation.
>
> **Key takeaways:**
> - Agent Threat Model is step zero: map out what the agent reads, calls, connects to, and where confirmation happens — before writing any defense
> - Tool Permission Isolation as 5 layers: read-only → user-read → draft → write → asset — each layer needs stronger guardrails
> - Prompt injection defense isn't about prompt engineering "don't listen to bad instructions" — it's structural: external data must never override system rules
> - Key/Secret Isolation principle: the agent requests a controlled action, never gets the secret itself
>
