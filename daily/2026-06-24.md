## 📖 Reading Path

### 🟢 Minimum Path (30 min) — 速读核心 + 项目自检

- [ ] **AI Security 速读 — 第一性原理 + 3 个与项目直接相关的节点** (25 min)
  - **第一性原理**（5 min）：「所有进入模型的内容都可能是攻击面，所有离开模型的动作都必须被约束。」Prompt injection 不只是「让模型说奇怪的话」——在 Agent 场景里，它可能诱导模型泄露密钥、修改交易目标、忽略风险检查、调用高权限工具。三条原则：上下文≠指令、工具权限要隔离、日志要可审计。
  - **Prompt Injection**（中级，10 min）：恶意内容可能藏在合约 README、网页内容、治理提案、token metadata、交易备注或外部 API 返回中。防护第一步：分层标注上下文——用户指令、系统规则、工具结果、网页内容、合约文档必须有不同可信级别。第二步：高风险工具不直接听模型一句话。**你的项目里，Parser 输出的是「parsed intent」，不是「executed transaction」——这本身就是 Prompt Injection 的防护。**
  - **Audit Log**（初级，5 min）：记录模型看到了什么、为什么选这个工具、policy 是否通过、用户是否确认。不能只记「最终回答」。**你的 CLI `history` 命令 + Day 37 设计的 v0.2 Audit Trail schema = 已经覆盖了这个节点。今天确认它是否被实装了。**
  - **Alert**（中级，5 min）：发现异常并及时打断。可监控信号：异常工具频率、预算快速消耗、非白名单地址、连续失败交易、session key 越界尝试。**你的 Budget 检查已经能拦截「超限」，但「异常频率」和「连续失败」的监控还没做——这是今天可以加的。**
  - **一句话连接**（5 min）：在 `## 📝 Learning Notes` 写一句——AI Security 的 8 个节点中，你的 AI-native Wallet 已经覆盖了哪几个（Audit Log ✅ / Prompt Injection 🟡 / Alert 🟡），最需要补哪几个（Sandbox / Permission Isolation / Tool Abuse 监控），最远的是哪几个（Key Safety——你已经在用 Session Key 而非主私钥，这已经是最好的 Key Safety 实践）。

### 🟡 Recommended Path (1 hour) — 精读全章 + 项目安全映射 + Security Model 初稿

- [ ] **AI Security 全章精读 — 8 个知识节点** (35 min)
  - 逐节点阅读并标注理解和项目关联：

  | 节点 | 难度 | 一句话理解 | AI-native Wallet 关联 |
  |---|---|---|---|
  | **Prompt Injection** | 中级 | 恶意上下文试图覆盖系统规则 | Parser 输出 parsed intent 而非 executed tx → 模型无法直接触发交易，这是第一道防线 |
  | **Tool Abuse** | 高级 | 滥用工具能力（反复调用付费 API、生成无限授权、调用非白名单合约） | Policy 的白名单 + Budget 的日/次/任务上限 = 工具滥用防线。但「循环调用」和「异常频率」还没有独立监控 |
  | **Malicious Context** | 中级 | 假合约地址、伪造审计报告、过期价格、被污染 token metadata | 你的 Parser 只解析用户自然语言，不直接读取合约文档/网页 → 当前攻击面小。但如果未来加 RAG 或网页爬取，就要标记 untrusted context |
  | **Key Safety** | 高级 | 私钥不进 prompt、不进日志、不进 analytics | ✅ 你已经用 Session Key 而非主私钥——这是 Key Safety 的最佳实践。主私钥从未进入 Agent runtime |
  | **Permission Isolation** | 高级 | 读工具/写工具/高权限工具分离，不给 Agent 万能 Web3 工具 | ✅ `policy.py` 的方法白名单（只允许 swap/transfer/approve）+ Session Key 的权限限制 = 工具隔离 |
  | **Sandbox** | 中级 | 隔离环境运行代码/浏览网页/处理文件 | ❌ 你的 Agent 当前在 CLI 环境运行，没有浏览器或代码执行需求——暂时不需要。但如果未来加网页爬取，就需要 sandbox |
  | **Audit Log** | 初级 | 记录完整链路：上下文→决策→工具调用→policy 判断→结果 | 🟡 CLI `history` = 雏形。Day 37 设计的结构化 Audit Trail schema 如果已实装 → ✅ |
  | **Alert** | 中级 | 异常频率/预算消耗/非白名单/连续失败/越界 → 触发响应 | 🟡 Budget 检查能拦截超限，但无「异常频率」或「连续失败」告警。需要 2-3 条告警规则 |

- [ ] **项目安全映射分析** (15 min)
  - 你的 AI-native Wallet 当前安全四层（Parser → Policy → Budget → Session Key + Human Check）vs AI Security 8 节点：

  | 你的防线 | 对应 AI Security 节点 | 状态 | 缺口 |
  |---|---|---|---|
  | Parser: NL→parsed intent | Prompt Injection | 🟡 | 只解析用户输入，不处理外部文档。未来加 RAG 时要标记 untrusted context |
  | Policy: 白名单合约/方法 | Permission Isolation + Tool Abuse | ✅ | 白名单限制了可调合约和可调方法 |
  | Budget: 日/周/任务/单次 | Tool Abuse（rate limit） | ✅ | 四维上限阻止超额消费 |
  | Session Key + Human Check | Key Safety + Alert | ✅+🟡 | Session Key 隔离主私钥 = ✅。但缺少异常频率告警 = 🟡 |
  | CLI `history` → Audit Trail | Audit Log | 🟡 | Day 37 schema 设计了但未确认实装 |
  | — | Malicious Context | ❌ | 当前无外部内容读取，攻击面小。预留标记机制 |
  | — | Sandbox | ❌ | CLI 环境无浏览器/代码执行需求。预留 |

- [ ] **Security Model 初稿** (10 min)
  - 基于 Day 36–39 四天阅读，在 Learning Notes 写一份 `## Security Model` 的 5 段文字初稿：
  1. **第一性原理**：「控制权不能交给概率系统。所有进入模型的内容都可能是攻击面，所有离开模型的动作都必须被约束。」
  2. **四层防线**：Parser（翻译不执行）→ Policy（规则拦截）→ Budget（额度限制）→ Session Key（最小权限）+ Human Check（分层确认）
  3. **攻击面分析**：当前攻击面（用户 NL 输入）→ 未来攻击面（合约文档/网页/API → 预留 untrusted context 标记）
  4. **可验证性**：Audit Trail 记录完整链路 → 每次操作可复盘、可争议、可改进
  5. **身份绑定**：Agent Identity 绑定 owner + session key → Agent 是可识别的经济参与者

  - **核心连接**：AI Security 防止攻击者伤害用户，AI Privacy 防止系统本身过度收集用户数据。两者是同一枚硬币的两面。
  - 重点关注 3 个与项目相关的节点：
    - **Data Boundary**（初级）：你的 AI-native Wallet 的数据边界是什么？用户输入 → Parser → LLM API → Policy → Wallet → 链上。哪些数据会发给 OpenAI？钱包地址？交易历史？
    - **Minimal Disclosure**（初级）：「证明余额足够」不需要暴露全部持仓。「模型只需要 parsed intent，不需要完整聊天历史。」
    - **Secret Management**（高级）：你已经在用 Session Key 而非主私钥 → 这是 Secret Management 的最佳实践。确认 Session Key 的私钥不会出现在任何 prompt 或日志中。
