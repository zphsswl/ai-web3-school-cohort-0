## 📖 Reading Path

### 🟢 Minimum Path (30 min) — 阅读核心节点 + 一句话连接

- [ ] **Agent Identity 速读 — 第一性原理 + 3 个初级节点** (25 min)
  - **第一性原理**（5 min）：Agent 身份必须绑定控制权、能力声明和服务入口，而不只是一个显示名称。身份要可解析、控制权要可证明、能力要可验证。核心是把 Agent 从临时会话变成可发现、可验证、可追责的经济参与者。
  - **Agent Profile**（初级，5 min）：Agent 的公开说明文件——名称、描述、服务范围、价格、接口、钱包地址、能力列表、model/工具说明、owner 和版本。关键洞察：Profile 要同时给人和机器看——人类读描述，机器解析 endpoint/capabilities/schemas/auth/payment/terms。**你昨天的 Agent Profile 设计已经覆盖了人类可读层，今天检查机器可读层。**
  - **Service Endpoint**（初级，5 min）：外部系统调用 Agent 的入口——可以是 HTTPS API、A2A endpoint、MCP server、Webhook。Endpoint 安全性直接影响身份可信度——如果 endpoint 被劫持，即使链上 agent id 没变，用户调用的也可能是恶意服务。**你的 AI-native Wallet 的入口是什么？CLI？API？还是两者？**
  - **Capability**（中级，5 min）：Agent 能完成什么任务，需要哪些输入和权限。关键原则：越具体越有用。每个 capability 要声明输入类型、输出格式、是否需要钱包权限、是否会调用外部 API、最长执行时间、失败如何退款。如果能力会触发链上动作，标记风险等级（只读=低、生成草稿=中、自动执行=高）。**你的 Agent 的能力声明在哪里？Policy 里的规则算不算能力声明？**
  - **Registry**（中级，5 min）：登记、发现和更新 Agent 身份。链上 registry 提供公开可查的身份锚点。它能证明"这个身份是谁注册的"，不能证明"这个 Agent 一定好用"。**你的项目用 GitHub repo 作 registry 吗？还是计划用 ERC-8004？**
- [ ] **一句话连接**（5 min）：在 `## 📝 Learning Notes` 写一句——Agent Identity 的 7 个节点中，你的 AI-native Wallet 已覆盖哪个（Profile 雏形？Endpoint？），最需要补哪个（Capability schema？Registry？Ownership 模型？），最远的是哪个（DID/VC？A2A？）。

### 🟡 Recommended Path (1 hour) — 精读全章 + 项目深度映射 + Profile 升级

- [ ] **精读全部 7 个知识节点** (35 min)
  - **Agent Profile**（5 min）：你已经做了人类可读的 Profile。今天补充机器可读字段——JSON schema 包含 name、description、version、owner_address、endpoint_url、capabilities[]、supported_chains[]、model_info、terms_url、privacy_policy、update_log[]。
  - **Capability**（5 min）：你昨天的 Profile 列了 3 个 capabilities（NL Intent Parsing / Transaction Drafting / Payment Execution），今天为每个补 risk_level、input_schema、output_schema、max_duration、failure_policy。
  - **Service Endpoint**（5 min）：定义你的 Agent 的 service endpoint——目前是 CLI `python main.py`，但对外暴露应该是什么？未来如果要让其他 Agent 调用，需要什么协议？
  - **Registry**（5 min）：理解链上 registry 的价值——ERC-8004 的 agent id / owner / profile URI / endpoint / 更新记录模型。你的 Agent 如果要在其他 Agent 或服务面前"证明自己是谁"，需要一个 registry 锚点。
  - **DID / VC**（5 min）：DID 提供跨平台身份，VC 承载能力证明、组织隶属、审计通过。核心洞察：VC 的可信度取决于 issuer——任何人都能签发声明 ≠ 任何声明都可信。产品要展示 issuer、签发时间、撤销状态。
  - **A2A**（5 min）：Agent 之间如何发现、通信、协商任务。A2A 消息最好和 Payment Intent、Receipt、Escrow 状态关联，否则对话和结算分裂成两套不可对账系统。**你的 Wallet 如果要被另一个 Agent 调用，A2A 是必须的。**
  - **Ownership**（5 min）：谁能更新 Agent profile、收款地址、服务 endpoint。高价值 Agent 不应该由单个热钱包控制。建议把 operator 和 owner 分开：operator 运行服务，owner 控制身份。**你的项目目前 owner = operator = 你的 EOA？未来如果需要多签或 DAO 控制怎么办？**
- [ ] **项目深度映射** (15 min)：用表格映射 7 节点：
  - Profile → 当前实现？gap？→ 你昨天的设计是 v0.1，今天补 schema
  - Capability → Policy 规则是否算能力声明？→ 不同：Policy 是约束，Capability 是声明
  - Endpoint → CLI 是 endpoint 吗？→ 对人是，对机器需要 API/A2A
  - Registry → GitHub = registry 吗？→ 临时方案，不够去中心化
  - DID/VC → 0 实现 → 最远但最通用
  - A2A → 0 实现 → 未来跨 Agent 协作必须
  - Ownership → EOA 单点 → 风险点，高价值后需升级
- [ ] **Agent Profile 升级** (10 min)：基于昨天的设计，补充：
  - JSON 机器可读版本（加在 profile 文件里）
  - 每个 capability 的 risk_level、input_schema、output_schema
  - Owner 和 operator 分离的设计说明
