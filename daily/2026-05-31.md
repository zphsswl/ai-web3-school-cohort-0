**Topic**: 智能体身份与信任 (Agent Identity & Trust)  
**Handbook**: https://aiweb3.school/zh/handbook/bridge/agent-identity/

Agent Identity isn't about giving an Agent a name — it's about making Agents **discoverable, verifiable, and accountable**. Once an Agent crosses application boundaries (pay for services, accept delegation, earn reputation), it needs provable identity: who owns it, what it can do, where to call it, and what its history looks like.

## 📝 Learning Notes

### Key Takeaways

1. **Agent Identity is NOT naming** — it's a bundle of: provable ownership, resolvable profile, verifiable capabilities, and service endpoint. Web3 provides the infrastructure for all four.

2. **Identity → Trust pipeline**: Identity (who) → Registry (discoverable) → Capability claims (what it can do) → Audit trail (what it did) → Reputation (should I trust it). Each layer builds on the previous.

3. **Where AI frameworks fall short**: LangChain/OpenAI agents use API keys for auth, but there's no standard way to publish capabilities, verify ownership cryptographically, or build cross-platform reputation. This is the gap Web3 fills.

### Bridge Connections

- **Day 14 (Machine Payment) → Day 15 (Agent Identity)**: Payment requires knowing *who* you're paying. Identity is the prerequisite for trustable machine payments.
- **Day 15 → Day 16 (Verifiable AI)**: Identity says "who you are" → Verifiable AI says "can I trust what you computed."

### Comparison: AI-only vs AI+Web3 Agent Identity

| Dimension | AI-only Agent | AI+Web3 Agent |
|-----------|---------------|---------------|
| Identity | Name + API key | DID + on-chain registry |
| Ownership proof | Login credentials | Wallet signature |
| Capability claims | README/docs | Signed VC claims |
| Service discovery | Manual/URL | Registry + A2A protocol |
| Reputation | Platform reviews | On-chain reputation/staking |
| Accountability | Platform TOS | Verifiable audit trail |
