**Handbook**: https://aiweb3.school/zh/handbook/tracks/wallet-permission/

**核心问题**: AI Agent 一旦能帮用户操作钱包，产品问题就会从「能不能完成任务」变成「它到底被允许做什么」。钱包不是一个签名弹窗，而是 Agent 执行能力的权限边界。

**第一性原理**: Agent 不应该拥有「钱包」，它只应该拥有一组可限制、可审计、可撤销的能力。把私钥交给 Agent 是最脆弱的设计 — 更稳的方式是让钱包/智能账户/权限模块持有资产，Agent 只提出意图、生成候选交易、请求有限权限。

**7 个知识节点**:

| # | 节点 | 难度 | 一句话 |
|---|------|------|--------|
| 1 | **AI Wallet UX** | 🟡 中级 | 不是把聊天框接到钱包，而是让用户知道 Agent 在做什么、为什么、下一步会发生什么 — 展示目标/计划/权限/交易解释/风险/撤销 |
| 2 | **Permission Policy** | 🟡 中级 | 写给执行层看的规则（资产范围、金额上限、合约白名单、函数范围、滑点、时间窗口、频率限制），必须可程序检查 |
| 3 | **Session Key Flow** | 🟡 中级 | 受限临时 key：用户创建→授权范围→Agent 使用→钱包校验→过期/撤销。减少重复签名但不能绕过风控 |
| 4 | **Safe Guard** | 🔴 高级 | 交易执行前检查器：目标地址白名单、calldata 函数检查、approval 额度、滑点、接收地址、模拟结果、频率 — 用确定性规则拦住不该发生的动作 |
| 5 | **ERC-4337 Workflow** | 🟡 中级 | 账户抽象 → UserOperation 流程：Agent 生成意图 → 包装 UserOp → Smart Account 校验 → Bundler 打包 → Paymaster 代付 → 链上执行 |
| 6 | **Pre-transaction Simulation** | 🟡 中级 | 签名前模拟：token 余额变化、授权产生、合约调用、失败可能、滑点/费用、无限授权风险 — 关键字段来自结构化解析和链上模拟 |
| 7 | **Recovery / Revocation** | 🟡 中级 | 权限仪表盘：查看所有 Agent/session key、资产/合约权限、已用额度、过期时间、历史动作、暂停/撤销/轮换/恢复入口 |

## 📤 Check-in Draft

> **Day 22 · Wallet / Permission (钱包与权限) · Phase 3: Frontier Exploration**
>
> Today explored the permission layer that makes AI Agent wallet operations safe — the core insight is that an Agent should never "own" a wallet, only possess a set of restrictable, auditable, and revocable capabilities.
>
> **Key takeaways:**
> - Wallet is not a signing popup but the permission boundary of Agent execution — it decides what the Agent can read, draft, and trigger
> - Permission Policy must be machine-checkable, not model-advised: asset scope, amount cap, contract whitelist, function scope, slippage, time window, frequency limit
> - Session Key Flow = temporary restricted key for low-risk high-frequency ops — reduces repeated signing but must not bypass risk controls
> - ERC-4337's UserOperation workflow is the infrastructure layer: Smart Account validates → Bundler submits → Paymaster sponsors → chain executes — permissions live in account logic, not model prompts
> - Pre-transaction Simulation is mandatory: token balance changes, approvals, contract calls, failure paths, unlimited approval risks — key fields from structured parsing + on-chain simulation, not LLM natural language
> - Recovery/Revocation dashboard: show all Agent keys, permissions, used quotas, expiry, recent actions, and pause/revoke/rotate entry points
>
