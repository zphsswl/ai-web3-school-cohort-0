**Handbook**: https://aiweb3.school/zh/handbook/tracks/agentic-commerce/

**核心问题**: 当 Agent 能代表用户发现商品、调用 API、比较报价、发起付款和验收结果时，商业流程应该如何被重新设计？**Agent 可以替用户发起商业动作，但不能替用户承担无限责任。**

**第一性原理**: 商业自动化的核心不是让 Agent 随便花钱，而是把购买意图、预算、验收标准和争议规则变成结构化协议。购买前要有 intent，购买中要有约束，购买后要有证据。

**6 个知识节点**:

| # | 节点 | 难度 | 一句话 |
|---|------|------|--------|
| 1 | **Agents Purchasing APIs** | 🟡 中级 | Agent 动态发现并购买 API/数据/推理服务 — 不是预注册，而是运行时服务发现+调用 |
| 2 | **Payment Intent** | 🟡 中级 | 结构化付款意图：目标、预算、支付资产、白名单、质量要求、退款条件、有效期 |
| 3 | **Budget Control** | 🟡 中级 | 五层预算：任务/服务/时间/风险/失败 — 防止 Agent 循环错误无限花钱 |
| 4 | **Proof of Task Completion** | 🟡 中级 | 不同服务的证据不同：API 响应+日志 / 数据哈希 / tx hash / 交付物 |
| 5 | **On-chain Receipt** | 🔴 高级 | 链上收据：payer/provider/task id/金额/输出哈希/验收状态 — 可追踪不可篡改 |
| 6 | **Escrow Flow** | 🔴 高级 | 托管状态机降低双向不信任 — 自动验收边界是高价值场景的核心挑战 |
