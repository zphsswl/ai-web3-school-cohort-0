Core takeaway: 链上数据是公开的，但不等于好用。Indexing 把区块、交易、事件和合约状态整理成产品、分析工具和 AI Agent 能快速查询的结构化数据。**链上是事实来源 (source of truth)，索引层是可用数据层 (usable data layer)。** RPC 不是数据库 — 你需要 Event Indexing、Subgraph、Data Pipeline 来构建面向问题的查询模型。

## 📝 Learning Notes

### 🧠 AI Core Concepts 补充 — 10 个 AI 核心概念速通

> 理解 AI × Web3 之前，先把 AI 的基础概念吃透。以下 10 个概念是构建 AI Agent 系统的基石。

---

#### 1. LLM（大语言模型）

**一句话解释**：LLM 就是大语言模型，能根据上下文理解问题并生成回答。

**具体例子**：比如我让 AI 总结一段代码，它会根据代码内容解释这段代码在做什么。

**常见误区或边界**：LLM 不一定说的都对，它有时会编造内容（幻觉），所以重要信息需要人工或资料复核。

---

#### 2. Prompt（提示词）

**一句话解释**：Prompt 就是我们给 AI 的任务指令。

**具体例子**：比如输入："请用表格对比三个 AI 客服产品的功能和优缺点。"这就是一个 prompt。

**常见误区或边界**：Prompt 不是越长越好，关键是要把任务、要求和输出格式说清楚。好的 prompt = 角色 + 任务 + 约束 + 输出格式。

---

#### 3. Context Window（上下文窗口）

**一句话解释**：Context window 是 AI 一次能看到和处理的上下文范围。

**具体例子**：如果上传一篇很长的论文，AI 可能不能一次完整理解所有内容，需要分段处理。

**常见误区或边界**：AI 不一定能一直记住所有历史内容（长对话中容易丢失中间信息），重要要求最好在关键任务里重新说明。

---

#### 4. Workflow（工作流）

**一句话解释**：Workflow 是把复杂任务拆成几个固定步骤，让 AI 按流程完成。

**具体例子**：做短视频脚本时，可以先拆解爆款结构 → 提取主题 → 生成新脚本 → 最后检查质量。

**常见误区或边界**：Workflow 更适合步骤明确的任务，不等于 Agent。它通常是「按流程走」，不是自己灵活决策。Workflow 是固定管道，Agent 是自主循环。

---

#### 5. Agent（智能体）

**一句话解释**：Agent 是能围绕目标自己判断、规划，并调用工具完成任务的 AI 系统。

**具体例子**：AI 客服可以识别用户问题 → 查询订单 → 判断是否转人工 → 给出处理结果。

**常见误区或边界**：Agent 不只是聊天机器人，它需要目标、工具、决策逻辑和安全限制。Agent = LLM + 工具 + 记忆 + 规划 + 执行循环。

---

#### 6. Tool Use（工具调用）

**一句话解释**：Tool use 是让 AI 调用外部工具，完成它自己不能直接完成的事。

**具体例子**：用户问订单物流，AI 可以调用订单系统查询状态，再告诉用户结果。

**常见误区或边界**：工具不是越多越好，只有在需要实时信息、准确数据或具体操作时才应该调用。工具列表过长会降低 AI 的决策质量。

---

#### 7. AI Coding（AI 编程）

**一句话解释**：AI coding 是用 AI 辅助写代码、改代码、查 bug 和生成测试。

**具体例子**：我可以让 AI 写一个 FastAPI 接口，也可以让它帮我分析前后端报错原因。

**常见误区或边界**：AI 写的代码不一定可靠，程序员仍然要检查逻辑、安全性和可维护性。AI 是副驾驶，不是自动驾驶。

---

#### 8. Guardrails（安全护栏）

**一句话解释**：Guardrails 是给 AI 设置规则和边界，防止它乱答或乱操作。

**具体例子**：AI 客服遇到退款、投诉、隐私问题时，不能直接承诺结果，而要转人工或生成工单。

**常见误区或边界**：Guardrails 不是一句「不要乱答」就够了，还需要规则、权限、工具限制和人工审核配合。多层次防护 > 单层限制。

---

#### 9. Tracing（链路追踪）

**一句话解释**：Tracing 是记录 AI 系统每一步做了什么，方便排查问题。

**具体例子**：如果 Agent 回答错了，可以通过 tracing 查看它识别了什么意图、调用了什么工具、得到了什么结果。

**常见误区或边界**：Tracing 不只是聊天记录，它更关注中间过程（意图识别 → 工具选择 → 结果处理），适合调试复杂 workflow 和 Agent。

---

#### 10. Human-in-the-loop（人机协同）

**一句话解释**：Human-in-the-loop 是在关键环节让人工参与确认或接管。

**具体例子**：AI 客服可以先收集信息，但遇到赔偿、投诉或隐私问题时，需要交给人工处理。

**常见误区或边界**：这不是 AI 能力弱，而是为了让系统更安全、更可靠。高风险任务（资金操作、敏感决策）通常需要人工参与。

---

### 🔗 与 Web3 的关联

| AI 概念 | Web3 中的场景 |
|---------|-------------|
| LLM | 理解链上数据、用户意图、生成交易摘要 |
| Prompt | 设计 Agent 的系统提示词，控制行为边界 |
| Context Window | 链上数据量大，需要分段/summary 放入上下文 |
| Workflow | DeFi 策略执行流程：监控 → 分析 → 决策 → 执行 |
| Agent | 链上交易 Agent：读取链状态 → 推理 → 签名 → 发送交易 |
| Tool Use | Agent 调用 RPC、合约 ABI、浏览器、价格预言机 |
| AI Coding | 用 AI 写/审/测智能合约代码（Solidity/Rust） |
| Guardrails | 限制 Agent 转账额度、禁止未经确认的合约调用 |
| Tracing | 追踪 Agent 的每笔交易决策链路（为什么执行这笔交易？） |
| Human-in-the-loop | 大额转账、合约部署、权限变更需人工确认 |

---

**Key Concepts / 核心概念：**

1. **LLM 是大脑，但不是全知全能的** — 有幻觉风险，需要验证
2. **Prompt 是方向盘** — 清晰指令 > 冗长描述
3. **Context Window 是视野** — 学会分段、摘要、重述关键信息
4. **Workflow ≠ Agent** — Workflow 是固定管道，Agent 是自主决策循环
5. **Agent = LLM + Tools + Memory + Planning** — 四个要素缺一不可
6. **Tool Use 要克制** — 只在需要外部数据或操作时才调用
7. **AI Coding 是副驾驶** — 代码要审查，安全要检查
8. **Guardrails 要分层** — 规则 + 权限 + 工具限制 + 人工审核
9. **Tracing 是黑匣子** — 出问题时靠它定位根因
10. **Human-in-the-loop 不是弱点** — 是安全关键设计

**Questions / 疑问：**

- AI Agent 在 Web3 中如何实现「自主但不失控」？Guardrails 的最佳实践是什么？
- Context Window 有限的情况下，如何让 Agent 高效处理链上大量数据？
- Web3 Agent 的 Tracing 和传统 Web2 应用监控有什么不同？
