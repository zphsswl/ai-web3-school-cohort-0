  - **第一性原理**（5 min）：「验证成本必须和输出影响力匹配」— 低风险留日志，高风险需 TEE/ZK/attestation。核心是把「相信模型」变成「验证证据和约束」。

### 🟡 Recommended Path (1 hour) — 精读 + 项目映射 + Audit Trail 设计

- [ ] **Verifiable AI 全章精读 — 7 个知识节点** (30 min)
  - 逐节点阅读并标注理解和项目关联：

  | 节点 | 难度 | 一句话理解 | AI-native Wallet 关联 |
  |---|---|---|---|
  | **TEE** | 中级 | 可信执行环境 + attestation 证明程序在受保护环境运行 | 你的 Agent runtime 在本地/服务器运行，暂不需要 TEE。但如果未来 Agent 作为服务部署，TEE 可证明「这个 Agent 确实在执行声明的 policy 代码」 |
  | **ZK** | 高级 | 证明计算满足条件而不泄露输入 | 暂时与你无关。但注意：Policy 检查是确定性的、计算量小的——如果未来需要链上验证「Agent 确实执行了 Policy 检查」，ZK 可以证明「规则被正确执行」 |
  | **zkML** | 高级 | 把 ML 推理转成可证明计算 | 最远。你的 Intent Parser（LLM）不需要 zkML。但混合方案思路值得记住：只证明关键步骤，不证明整个模型大脑 |
  | **Proof of Inference** | 高级 | 证明某输出来自某模型、某输入、某环境 | TEE / ZK / 签名日志 / 可重放推理都可实现。你的项目用签名日志路径（模型版本 + UserOp signature）已接近「轻量 Proof of Inference」 |
  | **Verifiable Compute** | 中级 | 链下计算，链上验证 | 你的 Policy + Budget 检查是链下执行的。理论上可以链上验证（但成本高）——当前阶段链下检查 + 链上 UserOp 签名是务实选择 |
  | **Benchmark** | 中级 | 不能替代任务级验证，需要自己的任务集 | 你需要一组攻击样本：swap 超限、错误 chain id、过期 quote、恶意合约地址。用这些测试 Policy/Budget/Session Key 是否真正拦截 |
  | **Audit Trail** | 初级 | 记录输入/输出/模型版本/工具调用/时间/用户确认/交易哈希/错误 | CLI `history` = Audit Trail 雏形。今天设计 v0.2 结构化方案 |

- [ ] **项目映射分析** (15 min)
  - 你的项目现状：
    - ✅ **Audit Trail 雏形**: CLI `history` 命令记录每次操作
    - ✅ **轻量 Proof of Inference**: 模型版本 + UserOp 签名 → 可追溯谁生成了什么交易
    - ✅ **Benchmark 意识**: Policy/Budget/Session Key 本身就是「边界测试」平台
    - ❌ **结构化 Audit Trail**: `history` 只是一行文本，没有结构化字段（input hash / model version / tool calls / user confirmation / tx hash / error code）
    - ❌ **攻击样本集**: 没有系统化的恶意测试用例
    - ❌ **Verifiable Compute**: Policy 检查是链下的，没有链上可验证性（当前阶段可接受）

- [ ] **Audit Trail v0.2 升级方案** (15 min)
  - 设计结构化 Audit Trail 的 JSON schema：

  ```json
  {
    "audit_id": "uuid",
    "timestamp": "ISO8601",
    "model": {"provider": "openai", "model": "gpt-4o", "version": "2024-08-06"},
    "intent": {"user_input_hash": "sha256(...)", "parsed_intent": "swap", "params": {...}},
    "policy_check": {"passed": true, "rules_checked": ["whitelist", "time", "method"], "duration_ms": 12},
    "budget_check": {"passed": true, "daily_used": 350, "daily_limit": 1000, "tx_amount": 50},
    "transaction": {"chain_id": 8453, "to": "0x...", "value": "0", "data_hash": "sha256(...)"},
    "session_key": {"key_id": "sk_abc123", "expires_at": "ISO8601", "permissions": ["swap", "transfer<100"]},
    "user_confirmation": {"required": false, "provided": false},
    "result": {"tx_hash": "0x...", "status": "success", "gas_used": 123456},
    "error": null
  }
  ```

  - 这个 schema 回答了 Verifiable AI 的核心问题：谁在什么时间、用什么模型、做什么操作、经过什么检查、产生了什么链上结果。
  - 不需要 TEE 或 ZK——纯日志已经提供了「可审计证据」。升级路径：v0.2 结构化日志 → v0.3 签名日志 → v1.0 TEE attestation。

- [ ] **精读 [Agent Identity](https://aiweb3.school/zh/handbook/bridge/agent-identity/) — 8 个知识节点** (35 min)
  - **Agent Profile**（初级）— Agent 的公开说明文件：名称、描述、能力、价格、接口、钱包地址、owner
  - **Capability**（中级）— 具体能力声明：输入/输出/schema/价格/限制/风险等级
  - **Service Endpoint**（初级）— 调用入口：HTTPS API / MCP server / A2A endpoint / 链上 registry
  - **Registry**（中级）— 登记、发现和更新 Agent 身份的链上/链下 registry
  - **DID / VC**（中级）— 去中心化身份 + 可验证声明：跨平台身份 + 能力证明
  - **A2A**（中级）— Agent-to-Agent 通信协议：发现、协商、委托、返回结果
  - **Ownership**（高级）— 控制权：EOA / Smart Account / 多签 / DAO，operator vs owner 分离
  - **最小实践**：设计一个 Agent Profile

- [ ] **你的 Agent Profile 设计** (20 min)
  - 为 AI-native Wallet 写一个 Agent Profile：
    - name: "AI-native Wallet v0.1"
    - owner: your EOA address
    - endpoint: CLI / future HTTPS API
    - capabilities: parse_intent / swap / transfer / check_balance
    - 每个 capability 标注风险等级：swap/transfer = 🔴 高风险，check_balance = 🟢 低风险
    - 写 3 行：谁可以更新 profile？更新后如何通知用户？如何证明这个 endpoint 确实属于该 Agent？

- [ ] **Verifiable AI + Agent Identity 融合方向** (15 min)
  - 写 100 字 v0.2 方向：
    - Agent Identity 给 AI-native Wallet 一个可解析的身份 → 其他 Agent 可以找到它、了解它的能力、验证它的 owner
    - Verifiable AI 给 AI-native Wallet 一个可追溯的执行记录 → 每次操作都有结构化 Audit Trail
    - 两者结合 = 你的 Agent 不仅「能做什么」是可验证的，「做过什么」也是可追溯的
    - Hackathon 角度：Agent Identity + Verifiable AI = AI Agent 的「信用报告」
